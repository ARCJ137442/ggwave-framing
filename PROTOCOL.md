# GGWave 分包协议规格 (GGWave-Fragmentation-Protocol v1.0)

> 版本：1.0 | 日期：2026-05-07 | 状态：设计完成

---

## 一、协议设计目标

### 1.1 设计背景

GGWave 单次传输限制：
- 最大数据：256 bytes
- 可变长度推荐上限：140 bytes
- 实际可用载荷：~95 bytes（留帧头和 ECC 空间）

### 1.2 设计目标

| 目标 | 说明 |
|------|------|
| 透明传输 | 应用数据原样传输，无损压缩 |
| 可靠传输 | 支持丢包检测和序号校验 |
| 简单实现 | 帧头尽量小，减少开销 |
| 向后兼容 | 可选的协议扩展字段 |

## 二、帧格式

### 2.1 数据帧结构

```
┌─────────────────────────────────────────────────────────────┐
│                    GGWave 分包协议帧                         │
├─────────────────────────────────────────────────────────────┤
│ [协议版本: 1B] [帧类型: 1B] [序号: 2B] [总帧数: 2B]      │
│ [数据载荷: N bytes]                                       │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 字段定义

| 字段 | 字节数 | 说明 | 约束 |
|------|--------|------|------|
| 协议版本 | 1 | 固定 0x01 | 未来兼容 |
| 帧类型 | 1 | 见帧类型表 | 必须 |
| 序号 | 2 | 当前帧序号（0-indexed） | 小端序 |
| 总帧数 | 2 | 总共多少帧 | 小端序 |
| 数据载荷 | N | 实际数据 | 最大 95 字节 |

### 2.3 帧类型

| 类型值 | 名称 | 说明 |
|--------|------|------|
| 0x01 | DATA | 数据帧 |
| 0x02 | EOF | 结束帧（最后一片数据） |
| 0x03 | ACK | 确认帧（预留） |
| 0xFF | SYNC | 同步帧（协议探测） |

### 2.4 示例帧

```
帧 0 of 210（开始）：
[01] [01] [00 00] [D2 00] [数据 95 bytes...]

帧 1 of 210：
[01] [01] [01 00] [D2 00] [数据 95 bytes...]

...

帧 209 of 210（结束）：
[01] [02] [D1 00] [D2 00] [数据 <95 bytes...]
```

## 三、协议流程

### 3.1 发送流程

```
应用数据（20,123 bytes）
    │
    ▼ 计算帧数
N = ceil(20123 / 95) = 212 帧（最后一帧不满）
    │
    ▼ 构建帧
for i in 0..N-1:
    frame = {
        version: 0x01,
        type: (i == N-1) ? EOF : DATA,
        seq: i,
        total: N,
        payload: data[i*95 : (i+1)*95]
    }
    │
    ▼ GGWave 编码
    audio = ggwave.encode(frame, AUDIBLE_FASTEST)
    │
    ▼ 写入 WAV
    wav.write(audio)
```

### 3.2 接收流程

```
WAV 文件
    │
    ▼ GGWave 解码
    frames = []
    for each audio segment in wav:
        frame = ggwave.decode(audio)
        if frame.version == 0x01:
            frames.append(frame)
    │
    ▼ 重组
    if all frames seq 0..N-1 received:
        data = concat(frames.payload)
    │
    ▼ 校验
    if len(data) == expected_length:
        return data
    else:
        return error
```

## 四、错误处理

### 4.1 丢包检测

| 检测方法 | 说明 |
|---------|------|
| 序号连续性 | 期望 seq[i+1]，收到 seq[i+2]，则 seq[i+1] 丢失 |
| 超时机制 | 超过 T ms 无新帧，认为丢包（需应用层实现） |

### 4.2 丢包处理策略

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| 重传请求 | 双向通信 | 发送 ACK 请求重传 |
| 跳过继续 | 单向传输 | 跳过丢包，继续接收后续帧 |
| 放弃 | 不可恢复 | 丢包过多，终止接收 |

**本实现采用「跳过继续」策略**：单向传输，无法请求重传，丢包则标记但不阻塞后续。

### 4.3 数据校验

| 校验项 | 方法 | 说明 |
|--------|------|------|
| 帧完整性 | 帧头 5 字节存在 | 版本+类型+序号+总数 |
| 序号有效性 | 0 <= seq < total | 防止越界 |
| 总数一致性 | 所有帧 total 相同 | 第一帧的 total 作为标准 |
| 数据长度 | 重组后长度校验 | 与总帧数×95 对比 |

## 五、参数选择

### 5.1 帧大小选择

| 参数 | 值 | 依据 |
|------|-----|------|
| 单帧最大载荷 | 95 bytes | 140 - 5(帧头) - 40(ECC余量) |
| 帧头大小 | 5 bytes | 版本1 + 类型1 + 序号2 + 总数2 |
| 每帧总大小 | ~100 bytes | 帧头 + 载荷 + ECC |

### 5.2 协议选择

| 协议 | 速率 | 时间(212帧) | 可靠性 |
|------|------|------------|--------|
| AUDIBLE_NORMAL | ~16 B/s | ~40 秒 | 高 |
| AUDIBLE_FAST | ~24 B/s | ~27 秒 | 中 |
| AUDIBLE_FASTEST | ~48 B/s | ~13 秒 | 低 |

**本实现采用 AUDIBLE_FAST**：平衡速度和可靠性。

## 六、局限性与未来改进

### 6.1 当前局限

1. **无重传机制**：单向传输，丢包无法恢复
2. **无压缩**：数据原样传输，可附加 gzip 压缩
3. **无加密**：可附加 AES 加密增强安全性

### 6.2 未来改进方向

| 改进 | 说明 | 优先级 |
|------|------|--------|
| ACK 重传 | 双向通信时支持重传请求 | P1 |
| gzip 压缩 | 减少传输时间和包数 | P2 |
| 帧校验和 | 每帧加 CRC-16 校验 | P2 |
| 流式编码 |边编码边写入，减少内存占用 | P3 |
